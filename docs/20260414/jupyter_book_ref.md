## 문서화 방식 비교: 전용 문서 레포 vs. 개발 레포 내 docs 폴더

다음은 두 가지 문서화 방식 —  
(1) 전용 문서 저장소 사용 (mybook1, mybook2, mybook3)  
(2) 개발 저장소 내 docs 폴더 사용 (repo1, repo2, repo3) —  
간의 주요 차이점을 정리한 내용입니다.

---

### 비교 표

| 항목 | 전용 문서 레포 방식 | 개발 레포 내 docs 방식 |
|------|---------------------|------------------------|
| 목적 | 문서 작성과 공유가 주 목적 | 코드 개발이 주 목적, 문서는 부수적 산출물 |
| 저장소 구조 | 문서 전용 저장소 | 개발 중심 저장소 |
| 폴더 구성 | 단일 문서 프로젝트 구조 | `src`, `experiments`, `notebooks`, `docs` 등 다중 폴더 |
| 문서 위치 | 저장소 루트 또는 전용 폴더 | `docs/` 폴더 내에 위치 |
| 배포 URL | `https://user.github.io/mybook1` | `https://user.github.io/repo1` |
| 빌드 명령 실행 위치 | 저장소 루트 | `docs` 디렉터리 내 |
| CI/CD 설정 | `jupyter-book build ./` | `cd docs && jupyter-book build ./` |
| 배포 디렉터리 | `./_build/html` | `./docs/_build/html` |
| 버전 관리 | 문서와 코드가 분리됨 | 문서와 코드가 동일한 저장소에서 관리 |
| 문서 유지보수 | 독립적 운영 가능 | 코드 변경과 함께 문서 업데이트 가능 |
| 협업 적합성 | 문서 전담 팀 또는 공식 가이드에 적합 | 개발자 중심 팀, 소규모 프로젝트에 적합 |
| URL 의미 | 문서 목적 중심 | 프로젝트 전체를 대표하는 통합 URL |

---

### 핵심 차이점 요약

1. **문서와 코드의 관계**  
   - 전용 레포: 문서와 코드가 분리됨. 문서는 독립된 자산으로 운영.  
   - 개발 레포 내 docs: 문서는 코드의 설명서 역할. 동일한 저장소에서 관리되어 동기화 용이.

2. **빌드 및 배포 프로세스**  
   - 전용 레포: 저장소 전체가 문서이므로 빌드 경로가 단순.  
   - 개발 레포 내 docs: CI/CD에서 명시적으로 `cd docs`로 이동 후 빌드 필요. 배포 경로도 조정 필수.

3. **CI/CD 워크플로우 예시**

   전용 레포:
   ```yaml
   - name: Build
     run: jupyter-book build ./
   ```

   개발 레포 내 docs:
   ```yaml
   - name: Build
     run: |
       cd docs
       jupyter-book build ./
   ```

4. **URL 전략**  
   - 전용 레포: 문서 목적 명확 (예: mybook1).  
   - 개발 레포: 저장소 이름과 동일한 URL (예: repo1), 프로젝트 통합성 강조.

5. **운영 복잡성**  
   - 전용 레포: 설정 단순, 문서 중심 운영에 유리.  
   - 개발 레포 내 docs: 다른 폴더와 혼재되므로 빌드 및 배포 경로 주의 필요.

---

### 선택 기준

| 상황 | 권장 방식 |
|------|----------|
| 공식 문서, 외부 공개용 가이드 | 전용 문서 레포 |
| 내부 실험 기록, 코드와 연동된 설명서 | 개발 레포 내 docs |
| 여러 프로젝트 통합 문서 사이트 | 전용 문서 레포 + 메인 인덱스 저장소 |
| 빠른 프로토타이핑 및 팀 내 공유 | 개발 레포 내 docs |

---

### 전환 가능성

- **개발 레포 → 전용 문서 레포**: `docs/` 내용을 새 저장소로 이전 후 CI 재설정.  
- **전용 문서 레포 → 통합 저장소**: 문서를 기존 저장소의 `docs/`로 이동 후 CI 조정.

---

### 결론

전용 문서 레포 방식은 문서의 독립성과 관리 용이성에 강점이 있습니다.  
개발 레포 내 docs 방식은 코드와 문서의 통합 관리가 가능하여 개발자 중심 워크플로우에 적합합니다.  
본 수정 가이드는 후자의 방식을 기반으로 하며, 개발과 문서화를 동시에 진행하는 환경에 최적화되어 있습니다.
